⏺ 配置分析

  你的配置有几个值得注意的点：

  1. authorization: disabled — 没有开启认证，线上环境建议开启
  2. bindIp: 127.0.0.1 — 只监听本地，安全性可以
  3. fork: true — 后台运行模式
  4. 看路径像是宝塔面板部署的 MongoDB

  ---
  一、排查宕机原因

  1. 查看 MongoDB 日志

  # 查看最近的错误日志
  tail -200 /www/server/mongodb/log/config.log

  # 搜索 OOM（内存不足）相关
  grep -i "oom\|killed\|out of memory" /www/server/mongodb/log/config.log

  # 搜索崩溃相关
  grep -i "fatal\|assert\|crash\|shutdown" /www/server/mongodb/log/config.log

  2. 检查系统层面

  # 查看系统日志，是否有 OOM Killer 记录
  dmesg | grep -i "oom\|killed\|mongo"

  # 或者
  journalctl --since "yesterday" | grep -i "oom\|mongo"

  # 查看内存使用情况
  free -h

  # 查看磁盘空间（磁盘满也会导致崩溃）
  df -h

  3. 常见宕机原因

  │           原因            │                   排查方式                   │

  ---
  二、设置自动重启

  方案一：systemd 服务（推荐）

  如果你的 MongoDB 是通过宝塔安装的，可能已经有 service 文件。检查一下：

  systemctl cat mongodb

  如果没有，创建一个：

  sudo vim /etc/systemd/system/mongodb.service

  [Unit]
  Description=MongoDB Database Server
  After=network.target

  [Service]
  Type=forking
  User=root
  ExecStart=/www/server/mongodb/bin/mongod --config
  /www/server/mongodb/config.conf
  ExecStop=/www/server/mongodb/bin/mongod --shutdown --dbpath
  /www/server/mongodb/data
  PIDFile=/www/server/mongodb/log/configsvr.pid
  Restart=always
  RestartSec=10

  [Install]
  WantedBy=multi-user.target

  ▎ 注意：确认 mongod 的实际路径，可以用 which mongod 或 ls 
  ▎ /www/server/mongodb/bin/

  然后启用：

  sudo systemctl daemon-reload
  sudo systemctl enable mongodb
  sudo systemctl start mongodb

  Restart=always + RestartSec=10 表示崩溃后 10 秒自动重启。

  方案二：宝塔面板直接设置

  宝塔面板 → 软件商店 → 找到 MongoDB → 设置 →
  通常有"自动重启"或"守护进程"选项，开启即可。

  方案三：crontab 定时检测（兜底）

  如果 systemd 不可靠，可以用 crontab 做健康检查：

  crontab -e

  添加（每 5 分钟检测一次）：

  */5 * * * * /www/server/mongodb/bin/mongosh --eval "db.adminCommand('ping')"
  --quiet 2>/dev/null || /etc/init.d/mongodb restart

  ---
  建议优先做的事

  1. 先看日志确认宕机原因 — tail -200 /www/server/mongodb/log/config.log
  2. 检查内存 — 如果是 OOM，需要加内存或限制 MongoDB 的 wiredTigerCacheSizeGB
  3. 用 systemd 托管并开启 Restart=always

  你可以先把日志最后 200 行贴出来，我帮你分析具体原因。

⏺ 原因确认：OOM Killer 杀掉了 MongoDB

  日志非常清晰：

  Out of memory: Killed process 38024 (mongod) total-vm:1992084kB,
  anon-rss:1379732kB

  MongoDB 占用了约 1.35GB 内存，服务器内存不足，被系统 OOM Killer 强制杀掉了。

  ---
  当前问题总结

  ┌──────────────┬───────────────────────────────────────────────────────────┐
  │     问题     │                           说明                            │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ OOM 杀进程   │ 服务器总内存不够，mongod 被系统杀掉                       │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ 两个服务冲突 │ mongodb.service（SysV/宝塔）和                            │
  │              │ mongod.service（systemd）同时存在，互相打架               │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ 日志文件权限 │ mongod.service 无法打开                                   │
  │              │ /www/server/mongodb/log/config.log                        │
  └──────────────┴───────────────────────────────────────────────────────────┘

  ---
  解决方案

  1. 限制 MongoDB 内存使用

  在你的配置文件中加上 WiredTiger 缓存限制：

  vim /www/server/mongodb/config.conf

  在 storage 部分添加：

  storage:
    dbPath: /www/server/mongodb/data
    directoryPerDB: true
    wiredTiger:
      engineConfig:
        cacheSizeGB: 0.5

  ▎ cacheSizeGB: 0.5 限制 MongoDB 最多用 512MB 
  ▎ 缓存。根据你服务器总内存调整，一般设为总内存的 50%-60% 减去其他服务占用。

  查看服务器总内存：

  free -h

  2. 清理冲突的服务，只保留一个

  # 查看两个服务的状态
  systemctl status mongodb
  systemctl status mongod

  # 禁用系统自带的 mongod.service（避免冲突）
  systemctl disable mongod
  systemctl stop mongod

  然后确保宝塔的 mongodb.service 有自动重启能力。

  3. 配置自动重启

  编辑宝塔生成的 SysV 启动脚本：

  cat /etc/rc.d/init.d/mongodb

  但更推荐的做法是创建一个原生 systemd 服务来替代。先确认 MongoDB 的路径：

  ls /www/server/mongodb/bin/mongod

  然后创建服务文件：

  vim /etc/systemd/system/mongodb.service

  [Unit]
  Description=MongoDB Database Server
  After=network.target

  [Service]
  Type=forking
  User=mongo
  ExecStart=/www/server/mongodb/bin/mongod --config
  /www/server/mongodb/config.conf
  ExecStop=/www/server/mongodb/bin/mongod --shutdown --dbpath
  /www/server/mongodb/data
  PIDFile=/www/server/mongodb/log/configsvr.pid
  Restart=always
  RestartSec=10
  LimitNOFILE=64000

  [Install]
  WantedBy=multi-user.target

  ▎ User=mongo 是根据日志中 USER=mongo 判断的，确认一下你的 MongoDB 运行用户。

  启用：

  systemctl daemon-reload
  systemctl enable mongodb
  systemctl restart mongodb
  systemctl status mongodb

  4. 修复日志文件权限

  chown mongo:mongo /www/server/mongodb/log/config.log
  chmod 644 /www/server/mongodb/log/config.log

  ---
  立即执行顺序

  # 1. 先看服务器内存
  free -h

  # 2. 限制 MongoDB 内存（编辑配置加 cacheSizeGB）

  # 3. 禁用冲突的 mongod.service

  # 5. 验证状态
  systemctl status mongodb

  # 1. 编辑配置文件

  systemctl status mongodb

⏺ 会有影响，主要是两个服务抢同一个端口和数据文件：

  ┌──────────────┬───────────────────────────────────────────────────────────┐
  │     风险     │                           说明                            │
  ├──────────────┼───────────────────────────────────────────────────────────┤

  趋势解释

  WiredTiger 缓存的特点是只增不减：

  1. 启动时缓存为空
  2. 每次查询/写入，数据页加载到缓存
  3. 缓存持续增长，直到达到 cacheSizeGB 上限（默认 ~1.3GB）
  4. 达到上限后通过 eviction（淘汰旧页）维持，但不会主动释放

  所以看起来是"一直增大"，其实是增长到上限后就稳定了，但默认上限太高，导致系统内
  存不够用。

  现在立即限制

  vim /www/server/mongodb/config.conf

  加上：

  storage:
    dbPath: /www/server/mongodb/data
    directoryPerDB: true
    wiredTiger:
      engineConfig:
        cacheSizeGB: 0.5

  然后重启：

  systemctl restart mongodb

  重启后再观察：

  ps aux | grep mongod

  预期 RSS 会降到 500-600MB 左右（缓存 512MB + 进程自身开销），OOM
  风险就解除了。
