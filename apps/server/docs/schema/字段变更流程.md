# MySQL 字段 / 表结构变更流程

上线与本地运行时，服务端以 **`apps/server/docs/schema/mysql-schema.json`** 为「期望中的完整表结构」：启动时若开启 **`autoSyncSchemaOnStartup`**，会对比当前库并按需执行 **`ADD COLUMN` / `MODIFY COLUMN`**。

以下步骤保证：新装机、自动对齐、管理面板漂移检测与生成的导出 SQL **四者语义一致**。

---

## 1. 约定：改哪里

| 文件                                  | 用途                                                                     |
| ------------------------------------- | ------------------------------------------------------------------------ |
| `docs/schema/mysql-schema.json`       | **唯一数据源**：漂移检测、`ADD/MODIFY`、管理端「预览建表 SQL」均由此生成 |
| `docs/sql/auto_create.sql`            | **空库 bootstrap / 手写建库**：应与 JSON 语义一致                        |
| `src/db/model/*.ts`（TypeORM Entity） | 运行时 ORM 映射，字段名、类型建议与 JSON 对齐，避免读写与库不一致        |

只改数据库而不改 JSON，下次对齐仍会把列改回期望；只改 Entity 不改 JSON **不会**再自动 MODIFY（当前逻辑不认 Entity 为准）。

---

## 2. 变更已有列（改类型 / 长度 / NULL / 默认值 / COMMENT）

1. **编辑 JSON**
   在 `mysql-schema.json` 中找到对应表 `tables[].columns[]`，修改该列的 **`ddl`** 片段（整条列定义须符合 MySQL 语法，与 `ALTER TABLE … MODIFY COLUMN` 后拼接一致）。
2. **同步 `auto_create.sql`**（强烈推荐）
   在 `CREATE TABLE` 与同表列列表中改掉同一列，并保持 `postStatements` 里与该列相关的 **`MODIFY`/索引**（若有）一致。
3. **同步 Entity**
   更新 `src/db/model/` 下对应 `@Column`，使 TypeScript 与库一致。
4. **上到环境**
   - 重启服务 **且** 配置项 **`autoSyncSchemaOnStartup`** 为开：启动时会按 JSON 做 **`MODIFY COLUMN`**（无变更时在日志中会输出「一致」）。
   - 或：管理面板 → 服务配置区域 →「一键对齐表结构」。
5. **看日志**
   - `[mysql:migrate] mysql-schema.json 与当前库一致（无 ADD COLUMN / MODIFY COLUMN）` — 无需变更。
   - `[mysql:migrate] 已变更：ADD COLUMN → …` / `MODIFY COLUMN → …` — 列出了本次改过哪些 `表.列`。

---

## 3. 新增列

1. 在 **`mysql-schema.json`** 的目标表 **`columns`** 末尾（或语义合适位置）增加一项：`name` + 完整 **`ddl`**（等价于 `ADD COLUMN` 后的片段）。
2. 同步 **`auto_create.sql`** 中该表的列定义。
3. 在对应 Entity 中增加属性与 `@Column`。
4. 重启（自动对齐开启）或在面板「一键对齐」— 服务端会 **`ADD COLUMN`**，日志同上「ADD COLUMN」行。

---

## 4. 新表或主键 / 索引 / AUTO_INCREMENT（复杂 DDL）

JSON 顶层 **`tables[]`**、`**postStatements**` 已与「导出脚本」对齐：

- **`tables`**：`CREATE TABLE IF NOT EXISTS` 的列 + `createOptions`。
- **`postStatements`**：`PRIMARY KEY`、`UNIQUE`、`MODIFY … AUTO_INCREMENT` 等后置语句。

改这类结构时需同时维护 JSON **`tables` + `postStatements`**，并同步 **`auto_create.sql`**。已有库一般不靠启动脚本「建表」，而是首次用 bootstrap SQL 或导出的整条脚本；运行时对齐仍以 **列级 ADD/MODIFY** 为主。

---

## 5. 手工执行 SQL（不重启或回滚预览）

管理面板：**预览建表 SQL（全屏）** — 内容由 `mysql-schema.json` **拼装**，可整段复制或框选片段在客户端执行。

---

## 6. 类型比对说明（为何会报「类型不符」）

对齐前会用 `information_schema.COLUMNS` 的 **`COLUMN_TYPE`** 与从 JSON **`ddl`** 解析出的期望类型做归一比较（整数显示宽度会被弱化比对）。若与 MySQL/MariaDB 实际类型字符串仍有边界差异导致误报，可优先调整 **`ddl` 写法** 与库里 `SHOW FULL COLUMNS` 保持一致。

---

## 7. 发布检查清单（建议）

- [ ] `mysql-schema.json`、`auto_create.sql`、相关 Entity **三处**已对齐
- [ ] 本地或预发：开 `autoSyncSchemaOnStartup` 启动一轮，日志无意外 `MODIFY`
- [ ] 管理面板 schema 概览 **无漂移**（或确认为预期的一次性对齐）
